# i18n

No external library. The i18n system is ~200 lines across four files and handles everything this app needs: locale detection, server and client APIs, dot-notation key lookups, parameter interpolation, RTL direction, and dynamic machine translation.

## How It Works

The active locale is stored in the `NEXT_LOCALE` cookie, set by middleware when a `?locale=` query parameter is present. Translation files live in `web/messages/` as JSON. The structure is namespace-keyed:

```json
// web/messages/en.json
{
  "Common": {
    "save": "Save",
    "cancel": "Cancel"
  },
  "Dashboard": {
    "title": "Dashboard",
    "greeting": "Welcome back, {name}"
  }
}
```

## Client API

Wrap components in `I18nProvider` (handled by `I18nLoader` in the root layout). Then use `useTranslations`:

```tsx
import { useTranslations } from "@/lib/i18n/provider";

function DashboardHeader() {
  const t = useTranslations("Dashboard");

  return (
    <h1>{t("title")}</h1>                          // "Dashboard"
    <p>{t("greeting", { name: "Alice" })}</p>       // "Welcome back, Alice"
    <p>{t("missing.key", undefined, "Fallback")}</p>// "Fallback"
  );
}
```

`useTranslations` accepts dot-notation namespaces: `useTranslations("Dashboard.Metrics")` resolves into the nested `Metrics` object under `Dashboard`.

Keys also support dot notation: `t("steps.first.title")` traverses the resolved namespace object.

## Server API

For server components and server actions, use `getTranslations`:

```ts
import { getTranslations } from "@/lib/i18n/server";

export default async function SettingsPage() {
  const t = await getTranslations("Settings");
  return <h1>{t("title")}</h1>;
}
```

`getTranslations` reads the `NEXT_LOCALE` cookie via `cookies()` and dynamic-imports the matching JSON file. Falls back to `en.json` if the locale file doesn't exist.

For static contexts (no cookie available), use `getStaticTranslations(namespace, locale)`.

## Adding a Locale

1. Create `web/messages/<locale>.json` with the same namespace structure as `en.json`
2. The system automatically picks it up — no config changes needed

Missing keys fall back to the key name itself (or an explicit default value if provided).

## RTL Support

`getDirection(locale)` in `web/lib/i18n/direction.ts` returns `"rtl"` for Arabic, Hebrew, Farsi, and Urdu; `"ltr"` for everything else.

The `LocaleSync` client component applies lang and dir to `document.documentElement` on mount. Include it in your root layout if you need HTML-level RTL:

```tsx
import { LocaleSync } from "@/lib/i18n/locale-sync";

// in layout.tsx
<LocaleSync locale={locale} />
```

## Dynamic Label Translation

`translateLabels()` in `web/lib/i18n/translate.ts` sends a flat labels object through the MyMemory free translation API. It's intended for merchant-defined UI labels (e.g. button text, announcement copy) that need to be translated at runtime.

```ts
import { translateLabels } from "@/lib/i18n/translate";

const translated = await translateLabels(
  { buttonText: "Add to cart", badge: "New" },
  "en",
  "fr"
);
// { buttonText: "Ajouter au panier", badge: "Nouveau" }
```

`{placeholder}` parameters are preserved during translation — the text is split on placeholder tokens, each segment translated independently, then rejoined.

The function maps short locale codes (`fr`, `de`, `ar`) to full BCP 47 tags (`fr-FR`, `de-DE`, `ar-SA`) before calling MyMemory. 40+ locales are mapped.

## Shop Locale Caching

Shop locales are fetched from Shopify via GraphQL and cached in `Shop.locales` (a `Json` field) with a 24-hour TTL. The `primaryLocale` field stores the shop's primary locale as a plain string for fast lookup.
